## 비즈니스 요구사항

### 핵심 규칙

- 매일 00:00에 해당 날짜의 상품이 공개된다.
- 하루 최대 10개의 기업만 참여할 수 있다.
- 하나의 기업은 하루에 상품 1개만 등록할 수 있다.
- 각 상품의 재고는 2~3개로 제한된다.
- 한 유저는 하루에 1개의 상품만 구매 또는 선점할 수 있다.
- 상품은 관리자 승인 후 판매 가능 상태가 된다.
- 결제 금액은 클라이언트 요청값이 아니라 서버에 저장된 상품 가격을 기준으로 검증한다.
- 재고 선점 후 10분 안에 결제를 완료하지 않으면 주문은 만료된다.


### 핵심 정책

### **3.1 판매 정책**

- 기업은 판매일 전 상품을 등록할 수 있다.
- 등록된 상품은 즉시 판매되지 않고 관리자 검수를 거친다.
- 관리자가 승인한 상품만 판매일에 공개된다.
- 매일 00:00에 해당 날짜의 승인된 상품이 ON_SALE 상태로 변경된다.
- 판매 종료 시점이 지나면 상품은 CLOSED 상태로 변경된다.
- 재고가 모두 소진되면 상품은 SOLD_OUT 상태가 된다.

상품 상태

```
READY      // 기업이 등록했지만 아직 판매 전
APPROVED   // 관리자가 승인함
ON_SALE    // 판매 중
SOLD_OUT   // 품절
CLOSED     // 판매 종료
REJECTED   // 관리자 반려
```

---

### **3.2 구매 정책**

- 한 유저는 하루에 하나의 상품만 구매 또는 선점할 수 있다.
- PENDING 상태의 주문도 당일 구매 제한에 포함한다.
- MVP에서는 한 번 주문을 시도하면 실패/만료/취소 여부와 관계없이 당일 추가 구매를 제한한다.
- 결제 제한 시간은 10분으로 한다.
- 10분 안에 결제가 완료되지 않으면 주문은 EXPIRED 처리되고 재고는 복구된다.
- 결제 완료 후 주문 상태는 PAID가 된다.
- 결제 완료 후 환불이 필요한 경우 REFUNDED 상태로 변경한다.

주문 상태

```json
PENDING    // 재고 선점, 결제 대기
PAID       // 결제 완료
FAILED     // 결제 실패
EXPIRED    // 결제 시간 초과
CANCELLED  // 사용자가 결제 전 주문 취소
REFUNDED   // 결제 완료 후 환불
```

주문 상태 흐름

```json
PENDING -> PAID
PENDING -> FAILED
PENDING -> EXPIRED
PENDING -> CANCELLED
PAID -> REFUNDED
```

---

### **3.3 재고 정책**

- 주문 생성과 재고 차감은 하나의 트랜잭션에서 처리한다.
- 재고 차감에 실패하면 주문은 생성하지 않는다.
- 재고가 0이면 주문 생성 요청은 실패한다.
- 결제 실패, 결제 만료, 주문 취소 시 재고를 복구한다.
- 재고 복구는 중복 실행되어도 한 번만 반영되어야 한다.
- 모든 재고 변경 이력은 StockHistory에 기록한다.

---

### **3.4 기업 등록 정책**

- 기업은 특정 판매일에 상품 1개만 등록할 수 있다.
- 하루 판매 슬롯은 최대 10개다.
- 상품 등록 시 DailySaleSlot을 함께 확보한다.
- 슬롯이 모두 찼으면 해당 날짜에는 상품을 등록할 수 없다.
- 승인 여부는 Product.status로 관리한다.
- 슬롯은 판매일 기준 기업 수 제한을 보장하기 위한 용도로 사용한다.

---

### **3.5 결제 정책**

- 클라이언트가 보낸 가격은 신뢰하지 않는다.
- 주문 금액은 서버 DB에 저장된 상품 가격으로 결정한다.
- 결제 승인 요청 시 서버의 order.total_price와 PG 승인 금액을 비교한다.
- 금액이 일치하지 않으면 결제 승인 실패로 처리한다.
- 주문 상태가 PENDING일 때만 결제 승인할 수 있다.
- 주문의 expired_at이 지난 경우 결제 승인할 수 없다.
