## 4. 예외 케이스 정의

| **상황** | **발생 가능한 문제** | **해결 방안** |
| --- | --- | --- |
| 동시성 충돌 | 재고 1개에 수백 명이 동시에 구매 요청 | DB 조건부 업데이트 또는 비관적 락으로 정확히 재고 수량만 차감 |
| 결제 중 이탈 | 재고를 선점했지만 결제를 완료하지 않음 | 주문 만료 시간 설정 후 EXPIRED 처리 및 재고 복구 |
| 중복 구매 시도 | 한 유저가 매크로로 여러 번 주문 요청 | user_id + sale_date 유니크 제약 및 서버 검증 |
| 기업 등록 경쟁 | 하루 10개 슬롯 초과 등록 발생 | 판매일 기준 슬롯 테이블 관리 및 DB 유니크 제약 |
| 결제 금액 위변조 | 클라이언트에서 가격을 조작해 결제 요청 | 서버 저장 가격과 PG 승인 금액 비교 |
| 결제 승인 실패 | 결제창에서는 성공처럼 보이나 서버 승인 실패 | 결제 상태를 FAILED 처리하고 재고 복구 |
| 서버 장애 | 결제 완료 후 서버 응답 실패 | 결제 승인 내역 재조회 및 주문 상태 보정 배치 |
| 새로고침/중복 클릭 | 같은 요청이 여러 번 전송됨 | 멱등성 키 적용 |
| 매크로 요청 | 비정상적으로 빠른 반복 요청 | Rate Limit, reCAPTCHA, IP/User 기반 제한 |
| 품절 후 요청 | 이미 품절된 상품에 계속 요청 발생 | 상품 상태를 SOLD_OUT으로 전환하고 빠른 실패 응답 |
